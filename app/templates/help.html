Steg 1

Opprett app-registrering i Microsoft Entra ID

{{ brand_name }} bruker app-only client credentials-flyten — altså en egen identitet for tjenesten, uten å låne en brukers kontekst. Dette er sikrest og krever ingen pålogging fra brukerne.

  1. Gå til entra.microsoft.com → Applications → App registrations.
  2. Klikk New registration. Gi den et navn, f.eks. {{ brand_name }} – {{ brand_owner }}.
  3. Velg Accounts in this organizational directory only. Hopp over Redirect URI.
  4. Etter opprettelse: noter ned Application (client) ID og Directory (tenant) ID.
  5. Gå til Certificates & secrets → New client secret. Velg utløpstid og kopier verdien med en gang — den vises bare én gang.
  6. Gå til API permissions → Add a permission → Microsoft Graph → Application permissions og legg til:
    • Files.Read.All
    • User.Read.All
    • Sites.Read.All (bare hvis dere vil indeksere SharePoint-områder)
  7. Klikk Grant admin consent for {{ brand_owner }}. Uten dette får app-only-kall 403.
Sikrere variant: Bruk Sites.Selected i stedet for Sites.Read.All og gi appen tilgang kun til spesifikke SharePoint-sider via en engangs PATCH mot /sites/{id}/permissions. Trinn-for-trinn i Microsofts dokumentasjon.
Steg 2

OpenAI-konto og API-nøkkel

  1. Logg inn på platform.openai.com.
  2. Gå til API keys → Create new secret key. Kopier verdien.
  3. Sjekk at kontoen har tilgang til text-embedding-3-large (alle paid-konti har det).
  4. Sett en månedlig usage limit i Limits som passer med antall dokumenter dere har.
Kostnad: ca. $0.13 per million tokens. 100 000 dokumenter på 5 sider hver ≈ 50 millioner tokens ≈ $6,50 én gang. Oppdateringer er marginale.
Steg 3

Pinecone-prosjekt og indeks

  1. Opprett konto på app.pinecone.io.
  2. Lag et nytt serverless-prosjekt (AWS, region us-east-1 er enklest).
  3. Hent API-nøkkel fra API Keys.
  4. Ikke opprett indeksen manuelt. Etter at miljøvariablene er satt, kjør: 
    python -m scripts.bootstrap_pinecone --cloud aws --region us-east-1
    Dette oppretter indeksen med nøyaktig riktig dimensjon (3072) og cosine-metric.

Filtyper

Hvilke filtyper indekseres — inkludert AutoCAD

{{ brand_name }} håndterer 25+ filtyper. Her er kategoriseringen:

Tekst og dokumenter (alltid)

  • Word: .docx, .doc, .odt, .rtf
  • Excel: .xlsx, .xls, .ods, .csv, .tsv
  • PowerPoint: .pptx, .ppt, .odp
  • Tekst og web: .txt, .md, .html, .xml, .json, .epub
  • E-post: .eml, .msg (Outlook)

PDF med OCR og bilder

PDF parses med Unstructured i hi_res-modus: layout-deteksjon identifiserer kolonner og tabeller, og scannet tekst leses med Tesseract OCR (norsk + engelsk). Innebygde bilder kan også embeddes via Gemini multimodal.

  • PDF: .pdf
  • Bilder: .png, .jpg, .tiff, .bmp

AutoCAD DWG og DXF

Egenutviklet CAD-pipeline for {{ brand_owner }}s tegningsarkiv:
  1. DWG (binær) konverteres til DXF (tekst) med libredwg-tools — installert i container.
  2. DXF parses med ezdxf som henter ut:
    • Drawing properties (tittel, prosjekt, forfatter, dato)
    • Titteltabell-attributter (KUNDE, PROSJEKT, REV, DATO, SKALA)
    • Alle TEXT, MTEXT, ATTRIB og MULTILEADER-elementer
    • Lagnavn (gir bygningsdel-taksonomi: A-WALL, M-PIPE-...)
    • Blokk-navn (symbol-bruk i tegningen)
    • Dimensjoner (mål)
  3. Tegningen rendres til en PNG som Gemini multimodal kan se direkte — du finner tegninger basert på visuelt utseende, ikke bare ord.

Hva som ikke støttes (ennå)

  • RVT (Revit) — proprietært binærformat, krever Revit selv eller Forge API
  • IFC (BIM) — kunne legges til; si fra hvis dette er viktig
  • Solidworks/Inventor 3D-modeller
  • Video-filer (men Gemini multimodal kan i prinsippet ta video-bilder)

Filer som hoppes over

Filer over MAX_FILE_BYTES (default 50 MB) hoppes over — store DWG-er kan trenge høyere grense. Også: filer som matcher SYNC_EXCLUDE_PATHS i innstillingene.

Integrasjon

Koble en ekstern LLM til {{ brand_name }} via MCP

Model Context Protocol (MCP) er en åpen standard fra Anthropic for å gi LLM-er tilgang til eksterne datakilder. {{ brand_name }} eksponerer kunnskapsbasen som en MCP-server, slik at verktøy som Claude Desktop, Cursor, Continue eller andre MCP-kompatible klienter kan søke i alle indekserte dokumenter uten egen integrasjon.

Hva får du?

Tre verktøy som klientene automatisk oppdager:

  • search_knowledge(query, top_k?, provider?, drive_id?, modality?) — vektorbasert søk i hele Byggkons kunnskap.
  • get_file_chunks(file_id, drive_id?, provider?) — hent alle deler av en spesifikk fil.
  • list_indexed_drives() — oversikt over hvilke OneDrives som er indeksert.

Slik kobler du til

  1. Sett MCP_BEARER_TOKEN i Railway-env-vars (eller via admin-UI). Lag en lang tilfeldig streng — denne er din "passord" for MCP-tilkobling. Eksempel: 
    openssl rand -hex 32
  2. Åpne /admin → fanen MCP-tilkobling.
  3. Kopier den klare-til-bruk-konfigurasjonen for klienten din (Claude Desktop, Cursor, eller Continue) og bytt ut placeholder-verdien med din faktiske MCP_BEARER_TOKEN.
  4. Restart klienten — verktøyene dukker opp automatisk.
Sikkerhet: Bearer-tokenet gir full lese-tilgang til kunnskapsbasen. Behandle det som et passord. Ikke commit det til Git eller lim det inn i delte chatter. Roter det ved å sette en ny verdi og redeploye — gamle klienter vil få 403 og du må oppdatere dem.

Eksempel — bruk fra Claude Desktop

Når Claude Desktop er koblet til, kan du si ting som:

  • "Søk i Byggkon-kunnskapen etter prosjekter med fuktskader på tak."
  • "Vis alle chunks fra rapporten 'Solås skole tilstand 2024'."
  • "Hvilke drives er indeksert?"

Claude vil automatisk kalle riktig search_knowledge-verktøy, få treff fra Pinecone, og bruke innholdet i sitt svar — med korrekte sitater og lenker tilbake til OneDrive-filene.

Feilsøking

Vanlige problemer

Graph-test feiler med 403 Forbidden

Du har sannsynligvis ikke gitt admin consent for application permissions. Gå tilbake til Entra ID → API permissions og klikk Grant admin consent.

Pinecone-test feiler med "index not found"

Du må kjøre python -m scripts.bootstrap_pinecone én gang for å opprette indeksen, eller du har skrevet feil indeksnavn i innstillingene.

OpenAI-test feiler med "model not found"

Sjekk at OpenAI-kontoen har text-embedding-3-large aktivert. Trial-konti har den ikke alltid; du må ha gjort minst én betaling.

Synkronisering kjører, men ingen filer indekseres

Sjekk SYNC_SCOPE. Hvis den er users_csv uten verdi i SYNC_USERS, blir det ingen drives å lese. Sett til all_users for å hente alle.

"Delta-token utløpt"-meldinger

Helt normalt etter lang stillstand. {{ brand_name }} resetter automatisk og kjører en full re-indeksering av den drivven på neste runde.

Containeren bruker mye disk

Imaget er ~3 GB pga. Unstructured + tesseract + libreoffice. Det er normalt for self-hosted dokumentparsing. Kjør hosted Unstructured API hvis dere vil ha mindre.
